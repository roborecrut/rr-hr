## Что делаем

ProTalk иногда возвращает пустой ответ (например, когда внутренний tool падает). Сейчас `ai-enhance` молча возвращает `{}`, и пользователь видит «успех» без изменений. Добавим защиту на бэке и понятную ошибку на фронте.

## Изменения

### 1. `supabase/functions/ai-enhance/index.ts`
- Для режимов `all_vacancy` / `all_company`: если `text` пустой или `tryParseJson` вернул `null`/пустой объект — сделать **1 ретрай** того же запроса с уточнением в system-промпте («верни ТОЛЬКО валидный JSON, без вызова инструментов, без поиска по URL»).
- Если и после ретрая ответ пустой → вернуть `{ error: "ai_empty_response" }` со статусом 502 (вместо `{ fields: {} }`).
- Для режима `single`: аналогично — если `text` пустой, ретрай; если снова пусто → 502 с `ai_empty_response`.
- Логировать оба попытки в `logs` через `logToDb` (чтобы видеть в дашборде).

### 2. `src/lib/aiClient.ts`
- `aiEnhanceAll` / `aiEnhanceSingle`: при ошибке от edge function пробрасывать понятное сообщение (`"ИИ не ответил, попробуйте ещё раз"` для `ai_empty_response`).

### 3. `src/components/AIWaitProvider.tsx` (по необходимости)
- Убедиться, что при throw из `task()` оверлей показывает состояние ошибки (грустный RR9) с кнопкой «Повторить», а не «успех». Если уже так — не трогаем.

### 4. Места вызова `aiEnhanceAll` (например `VacancyEditor.tsx`, `CompanySections.tsx`, и где ещё используется)
- Проверить, что после `aiWaitRun(...)` если результат пустой/ошибка — не перезаписываем поля пустыми значениями. Уже есть `if (!result) return;` — оставляем, но дополнительно: если `Object.keys(result).length === 0` — тоже return + toast «ИИ не вернул данные».

## Что НЕ трогаем
- Конфиг бота в ProTalk (это уже сделал пользователь).
- Логику `/restart` и других AI-вызовов — только enhance.

## Технические детали
Ретрай в edge-функции — простой `for (let attempt = 0; attempt < 2; attempt++)` вокруг `callProTalk`, со вторым system-промптом построже. Таймаут оставляем 110с.
