## Итерация 1 — Критичные баги (П.6, страховка по П.5)

**Цель:** Список вакансий снова виден в `/empXXXXX/vacancies`, кабинет работодателя не зависит от несуществующих `/api/*` ручек.

### Что делаем

1. **EmployerPanel.tsx — выкинуть фантомные `fetch("/api/...")`.**
   В Vite-SPA эндпоинты `/api/projects`, `/api/projects/:id`, `/api/companies`, `/api/employers/me` и т.п. не существуют. Dev-сервер отдаёт HTML 200 → `res.ok === true` → `res.json()` падает SyntaxError → внешний try/catch проглатывает → списки пустые. Заменяем на прямые запросы к Supabase (fallback-блоки уже есть).

2. **Подчистка `fetchData` и связанных мутаций:** удалить ветки `if (res && res.ok)` для проектов, кандидатов, компаний, профиля; оставить только supabase-ветки. Сохранение/удаление проекта тоже идёт через `/api/projects/:id` — заменить на `supabase.from('projects').update/delete`.

3. **Проверка п.5 (страховка).** Пользователь пишет, что прошло само (вероятно кэш Google OAuth). Добавим:
   - В `signOut` хедера AdminPanel и EmployerPanel — `await supabase.auth.signOut({ scope: 'global' })` + чистка `localStorage.employer_session_id` и `candidate_session_id`.
   - На странице `/` (или общем `SessionBootstrap`) — если `getSession()` вернул валидную, но в `user_roles` нет ролей и нет привязки employer/candidate, не блокируем ввод другим аккаунтом.

4. **Проверка после фикса:** открыть `/emp100003/vacancies` в превью — должна появиться вакансия 400002.

### Технические детали
- Файлы: `src/pages/EmployerPanel.tsx` (только удаление мёртвых веток + supabase-фолбэк уже написан), `src/pages/AdminPanel.tsx` (signOut с `scope: 'global'`).
- Без миграций БД.

## Итерация 2 — Мастер создания вакансии + 15 полей (П.1, П.2)

**Цель:** при создании новой вакансии работодатель сначала выбирает компанию и должность, и только потом видит уже предзаполненные поля. AI знает про все 15 полей и их формат.

### Что делаем

#### А. Миграция БД — 2 новых поля
Добавить в `public.projects` (и при необходимости в `public.training_blocks`):
- `onboarding_text TEXT`
- `motivation_text_detail TEXT`

Шаблоны в `job_titles.field_templates` уже хранят их в JSONB — миграции для шаблонов не нужно.

#### Б. Шаг 0 в мастере вакансии — выбор компании и должности

Новый экран перед формой:
```
┌─────────────────────────────────────────┐
│ Шаг 1/2 — Выберите компанию             │
│ [▾ Список компаний работодателя]        │
│ [+ Создать новую]                       │
├─────────────────────────────────────────┤
│ Шаг 2/2 — Выберите должность            │
│ ┌──────────────────────────────────┐    │
│ │ 🔍 Введите или выберите...       │    │
│ └──────────────────────────────────┘    │
│ ▸ Менеджер по продажам    [шаблон]      │
│ ▸ Оператор колл-центра    [шаблон]      │
│ ▸ Кассир                  [шаблон]      │
│ ... (а-я, фильтр по вводу)              │
│                                          │
│ [Продолжить с этой должностью]          │
└─────────────────────────────────────────┘
```

Логика:
1. При открытии — `supabase.rpc('admin_list_job_titles')` НЕ годится (только админ). Делаем публичный RPC `job_titles_list_public()` → возвращает `id, title, is_basic, usage_count` (без `field_templates` чтобы не светить весь JSON). Сортировка по `title ASC`.
2. Поиск-фильтр на клиенте (массив ≤500 элементов — не нужен серверный поиск).
3. **Если выбрана из списка** → `job_title_get_templates(_title)` → предзаполнить все 15 полей вакансии.
4. **Если ввели уникальную (нет в списке)** → 
   - сначала попытаться загрузить шаблон «Менеджер по продажам» как базу,
   - параллельно запустить ProTalk (edge function `ai-enhance` с режимом `bulk_template`) — она вернёт черновик на 15 полей по введённой должности и заменит дефолт.
5. Только после этого пользователь видит обычную форму редактирования вакансии со всеми полями.

#### В. 15 полей везде

Обновить массив `FIELDS` в:
- `src/pages/EmployerPanel.tsx` (мастер вакансии)
- `src/components/TrainingWizard.tsx`
- `src/pages/AdminPanel.tsx`
- `src/components/VacancySections.tsx` и `CompanySections.tsx` (рендер на лендинге)
- `src/lib/vacancyTemplates.ts`

Новые поля:
- `onboarding_text` — группа «Вакансия», блок «Этапы трудоустройства»
- `motivation_text_detail` — группа «Вакансия», блок «Мотивация — детально» (рядом с `motivation_text`)

#### Г. AI знает формат всех 15 полей

В `src/lib/aiClient.ts` → `aiEnhanceSingle`: добавить параметр `formatExamples` — словарь `{ field_key: пример_текста_формата }`. Передавать в edge-функцию `ai-enhance` как часть system prompt:

```
ВАЖНО: каждое поле имеет свой формат. Ниже эталоны:
- payouts_text: "Оклад X ₽ + %... Выплаты N и N числа..."
- vacancy_text: маркированный список из 4-6 пунктов через •
- tasks_activity_text: пункты с эмодзи в квадратных скобках: "• [📞 Консультация] ..."
- system_text_vac: пункты с тегом-категорией: "• [CRM] Bitrix24 — ..."
... (все 15)
Текущее поле: {field}. Соблюдай его формат строго.
```

Эталоны кладём в новый файл `src/lib/fieldFormats.ts` (один источник правды, импортируется и клиентом, и edge-функцией через прямую константу).

### Технические детали
- Миграция: `ALTER TABLE projects ADD COLUMN ... ; ALTER TABLE training_blocks ADD ...` (если нужно).
- Новый RPC `job_titles_list_public()` — STABLE, `GRANT EXECUTE TO authenticated, anon`.
- Edge function `ai-enhance` — расширяется опциональным `format_examples` в body.
- Новый компонент: `src/components/VacancyCreationWizard.tsx` (шаг 0). Существующий мастер становится «шагом 1».

## Итерация 3 — Обучение в 3 этапа + синхронизация кабинета кандидата (П.4, П.7)

**Цель:** обучение всегда показывает 3 этапа (база / продукт / системы), наполняясь из шаблона, AI-генерации или загруженных документов. Кабинет кандидата рендерит вакансию и обучение точно так же, как публичные лендинги.

### П.4 — структура курса

Сейчас `TrainingCoursePreview.tsx` рендерит блоки динамически. Делаем:

1. Жёсткий каркас из 3 разделов:
   - **«Профессиональное обучение»** ← поля `training_professional_text` + AI-сгенерённые уроки этой группы
   - **«Продукт и компания»** ← `training_product_text` + загруженные клиентом материалы
   - **«Работа в системе»** ← `training_systems_text` + `training_wiki_text` + `training_regulations_text`

2. Если у вакансии есть AI-сгенерированные уроки (`training_lessons` через `ai-generate-onboarding`) — распределяем их по 3 этапам по полю `category` (добавить такое поле, default = 'professional').

3. Если у клиента загружены документы/регламенты (`projects.custom_wiki`, аплоады) — добавляем как дополнительные материалы в «Работа в системе».

4. Введение (`training_intro_text`) — общий заголовок сверху.

### П.7 — единый рендер

Сейчас кабинет кандидата (`CandidateFlow.tsx`) частично дублирует разметку лендингов. Делаем:

1. Экстрагируем `VacancySections` и `CompanySections` в чистые презентационные компоненты с props `{ project, company }` (без логики загрузки).
2. В `CandidateFlow` и в `JobVacancyLanding/CompanyLanding` импортируем одни и те же компоненты.
3. Источник данных — всегда Supabase (`projects` + `companies` + `training_blocks`), никаких локальных дефолтов.
4. Аналогично для `TrainingCoursePreview` — общий компонент между лендингом и кабинетом кандидата.

### Технические детали
- Миграция (если делать категоризацию уроков): `ALTER TABLE training_lessons ADD COLUMN category TEXT DEFAULT 'professional'` + чек `category IN ('professional','product','systems')` через триггер.
- Рефакторинг без новых RPC.
